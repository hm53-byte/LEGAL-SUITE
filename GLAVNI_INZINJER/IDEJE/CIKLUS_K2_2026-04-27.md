## Izazov

Kako uvesti **offline-first PWA** (Service Worker + IndexedDB + Web Crypto API + manifest) u APLIKACIJA stack tako da:

1. **Streamlit ostaje deterministic backend**: 60+ generator funkcija u `generatori/*.py` ostaju Python pure-functions koje produciraju isti docx za isti input (preduvjet K1).
2. **K1 audit chain ostaje cjelovit**: forme ispunjene offline → zapisane lokalno → kasnije sinkronizirane na server **ne smiju razbiti** SHA256 hash chain u `download_log` (parent_hash → current_hash).
3. **Korisnik može raditi bez internetа**: aplikacija + zadnje forme dostupne lokalno; auto-save spriječava gubitak rada pri browser crash-u.
4. **Privacy poboljšana**: korisnikovi privatni podaci (OIB-i, klauzule) opcijski enkriptirani u IndexedDB s ključem deriviranim iz lozinke (server ne vidi plaintext kad korisnik tako odluči).

Specifični use case: pravnik radi u sudnici bez interneta, ispunjava tužbu, sprema lokalno; vraća se u ured, klikne "sync" — server prima pohranjeni input, generator se pokreće, docx se producira **s ispravnim K1 audit linkom** (parent_hash je posljednji hash u korisnikovom lancu, current_hash slijedi).

## Što NIJE u scope-u (Pravilo L6 anti-AI drift + L4 ROI fokus)

- ❌ Client-side docx generacija u JS (port 60+ generator funkcija na JS razbija deterministic guarantee — različit kod = različit `generator_version_hash`, K1 lanac postaje multi-stack mess)
- ❌ Service Worker koji "predviđa" što korisnik radi (no AI inference)
- ❌ Smart auto-complete iz povijesti (recommender = AI Act flag)
- ❌ Sync conflict resolution kroz "AI suggester"
- ❌ "Smart" cache invalidacija temeljena na klasifikaciji content-a

Sve gornje aktiviraju AI Act zone ili razbijaju determinizam. K2 mora ostati **isključivo deterministic infrastrukturni sloj** preko postojećeg server-side koda.

### Pravilo L6 — Što ne smijemo dirati

K1 invarianta: `parent_hash → current_hash` lanac u `download_log`. Ako offline mod producira docx koji preskoči chain (jer je server bio nedostupan), **dva problema**:

1. Drugi korisnici nastavljaju lanac; offline korisnik vraća se i njegov entry treba parent_hash koji je u međuvremenu napredovao
2. Order delivery: dvije offline forme od istog korisnika trebaju ispravnu kronologiju

K2 mora **delegirati hash chain construction serveru** — klijent NE računa parent_hash sam (jer ne zna trenutno stanje servera u trenu sync-a). Klijent samo šalje `(input_canonical_form, doc_type, generator_module_path, client_timestamp, client_event_id)` u outbox; server pri primanju računa hash chain link. Ovo je arhitekturno čisto.

Pokrenut preko `Agent` tool-a (general-purpose, prazan context). Subagent vratio 5 kristala u ~24k tokena. Kineski zid intaktan (provjereno: nigdje nije spomenuto "softver", "PWA", "IndexedDB", "Service Worker", "K1", "audit", "APLIKACIJA", "sync"; jedinstvena granica je riječ "hash" u K2 mehaničkom kosturu — koristi se kao generic matematički termin za "integrity check function", ne kao SHA256 referenca; akceptirano).

### KRISTAL 1: KOMPLETNI FRAGMENTI BEZ MOSTA (Calvino)

- Izvor: Italo Calvino, *Se una notte d'inverno un viaggiatore* (1979)
- Apstraktni princip: Sustav isporučuje niz cjelovitih početaka koji se ne nastavljaju u jedinstveni produkt; sintezu konstruira primatelj presijecanjem fragmenata, a ne autor zatvaranjem nastavka.
- Mehanički kostur: tri koraka — (1) generiraj fragment koji je interno koherentan (početak + srednji dio + prekid), (2) prekini u trenutku najveće informacijske gustoće, (3) ponudi sljedeći fragment iz drugog konteksta. Stanje: {N otvorenih fragmenata, 0 zatvorenih lanaca, M tragova povezivanja primatelja}. Uvjet prijelaza: strukturna pozicija (prekid u istoj fazi), ne semantička srodnost.
- Diskriminantni uvjet: izmjeri prosječnu duljinu fragmenta i dispersiju točke prekida; ako >80% završi prirodno, princip ne važi; ako primatelji ne grade veze (0 referenci), to je samo zbirka.
- ERROR/FIX (4): predugi fragmenti gube tenziju; klikbejt fragmenti frustriraju, ne aktiviraju; primatelj bez kapaciteta = mrtvi fragmenti; razlika od "to be continued" — ovdje nastavak NE postoji.

### KRISTAL 2: GRANIČAR TRIJU KRALJEVSTAVA (Hermes)

- Izvor: Hermes, grčka mitologija (psihopomp, granicar)
- Apstraktni princip: Postoji uloga čija je jedina funkcija prijenos sadržaja preko granice između sfera koje inače međusobno ne komuniciraju, uz uvjet da se sadržaj ne deformira u prijenosu.
- Mehanički kostur: ≥3 odvojene sfere s vlastitim pravilima; granicar ima dva strogo razdvojena moda — (1) translacija (preformuliraj u jezik ciljne sfere), (2) preservacija (drži semantiku invarijantnu); uvjet integriteta: provjerljiva ekvivalencija ulazne i izlazne poruke; stanje: {trenutna sfera, nosi=poruka, integritet=hash}.
- Diskriminantni uvjet: definiraj P na izvoru, izmjeri P' na odredištu, izračunaj d(P, P'); ako > prag, granicar zatajio. Ako sfere nisu ontološki odvojene, princip suvišan.
- ERROR/FIX (4): granicar-postaje-tumač = integritet pada; nije svaki API "Hermes" (granica mora biti ontološka); zajednički meta-jezik = redundanca; jednosmjerni prijenos nije Hermesov.

### KRISTAL 3: FORSIRANA OKLADA (Pascal)

- Izvor: Pascal, *Pensées* fr. 233 (1670)
- Apstraktni princip: Pod fundamentalnom neizvjesnošću o stanju svijeta, neodlučivanje nije neutralna pozicija — ono je već jedan od izbora s vlastitim posljedicama; matrica odluke obvezna, ne fakultativna.
- Mehanički kostur: stanje S ∈ {S0, S1} nepoznato, agent bira A ∈ {a0, a1}; postoji četverokutna matrica U(A,S); treća opcija "ne biraj" mapira se na default-akciju, ne stoji izvan; postupak: identificiraj default-akciju → izračunaj očekivani ishod → obvezno odluči.
- Diskriminantni uvjet: ako agent ne odluči u T → mjerljiva posljedica; ako sustav stoji u istom stanju zauvijek, princip ne važi i postoji prava treća opcija.
- ERROR/FIX (4): zlouporaba kao retorički trik gdje vrijeme stvarno ne pritišće; pesimistična manipulacija preko nepoznatih ishoda; kontinuirani S degradira binarnu okladu; pozitivna vrijednost čekanja čini odgodu racionalnom.

### KRISTAL 4: TROFAZNI PRAG BEZ POKRIĆA (Turner)

- Izvor: Victor Turner, *The Ritual Process* (1969)
- Apstraktni princip: Promjena statusa subjekta nije skok, nego trofazna trajektorija s eksplicitno strukturiranim međufaznim stanjem u kojem subjekt nema pokriće ni starih ni novih pravila.
- Mehanički kostur: stanja {S_old, S_liminal, S_new}; tranzicije — (1) separacija (eksplicitan rez, sustav registrira), (2) liminalnost (privremena pravila, snižena zaštita), (3) reintegracija (eksplicitno priznanje S_new); trajanje liminalne faze nije nula i nije beskonačno; mora biti omeđeno.
- Diskriminantni uvjet: postoji li mjerljiv prelazak kroz srednje stanje (drukčija pravila), ili instant state-change. Postoji li eksplicit početak i kraj liminalne faze.
- ERROR/FIX (4): permanentna liminalnost = communitas se rutinizira (treba timeout); nije svaki "tranzicijski period" Turner (mora imati vlastita pravila); subjekt koji ne prihvaća separaciju = nelegitiman proces; asinhronost (oba pravila istovremeno) razbija princip.

### KRISTAL 5: PRIVATNI SLOJ PRIJE JAVNOG (Lichtenberg)

- Izvor: Lichtenberg, *Sudelbücher* (1765-1799)
- Apstraktni princip: Asimetrični dvoslojni zapis — primarni (ulazni) sloj prima sve bez filtera, sekundarni (objavljeni) sloj prima samo manjinu nakon eksplicitne selekcije; većina sadržaja nikad ne prelazi i to je strukturna značajka.
- Mehanički kostur: dva spremnika {wastebook W, ledger L}; pravila — (1) zapis u W jeftin, neuredan, bez sheme, (2) prijelaz W→L zahtijeva eksplicitan korak selekcije, (3) većina W ostaje u W zauvijek; očekivana stopa promocije < 50%; |W| raste, |L| sporo, omjer stabilan.
- Diskriminantni uvjet: stopa promocije; ako >70% nije Lichtenberg (samo "draft → final"); ako =0%, W je smetlište; aktivna selekcijska rutina nužna ALI većina ostaje u W.
- ERROR/FIX (4): javan W ukida dvoslojnost; nije svaki "drafts folder" Lichtenberg (W mora biti čitljiv sam za sebe); skup upis u W = autocenzura kolapsira sustav; brisanje W ukida posmrtnu rekonstrukciju.

### Najjači spoj (Pravilo L5: 1 kandidat, depth > breadth)

**K2 + K3 + K4** daju arhitekturni okvir za **deterministic offline outbox sync**:
- Hermes (K2) je SW kao border guardian s preservacijom semantičkog payload-a
- Pascal (K3) je idempotency key kao "forsirana oklada" pri submitu — odluka se ne može povući, ali idempotentnost čini retry siguran
- Turner (K4) je three-state sync FSM s timeout-om za liminalnu fazu

**K1 (Calvino) je preduvjet** — forme se serijaliziraju kao kompletne samostalne pakete; server ne dobiva fragment, dobije gotov input.

**K5 (Lichtenberg) je orijentacija privatnosti** — IDB outbox je wastebook, Supabase je ledger; korisnik ima kontrolu nad promocijom.

**Što EKSPLICITNO odbacujem (kontradikcija s K1 Janusovim audit lancem)**:
- Client-side docx generacija u JS — port 60+ generator funkcija na JS razbija deterministic guarantee i mijenja generator_version_hash kontroliran s `generators_registry.json`. Generacija OSTAJE server-side.
- Klijent-side hash chain link computation — klijent NE zna trenutni `parent_hash` u trenu submita (server može imati novije zapise od drugih sesija); chain link **mora** biti server-side.

**Web Crypto enkripcija (K5 / §II.3)** — opcijska faza 2; MVP ne zahtijeva (privatnost je već poboljšana time što se input ne šalje na server dok korisnik ne klikne sync). Faza 2 dodaje AES-GCM enkripciju IDB outbox entries.

## PhD-tehnička pitanja (Pravilo 16, 4/5)

1. **Distribution shift**: relevantan u **client-side state shift-u** — različiti browseri (Chrome 120, Firefox 130, Safari 17) imaju različitu IDB implementaciju; Chrome ima najveću kvotu (~60% disk), Safari samo 1GB. Mitigacija: feature detection + `navigator.storage.estimate()` warning ako kvota < 100MB; ne uskočiti u IDB ako nije dostupan (graceful fallback to no-offline mod).

2. **Calibration**: SW cache strategija mora biti **kalibrirana** prema asset tipu — JS/CSS/HTML su safe za cache-first (immutable s hash u URL-u); API responses MORAJU biti network-first (uvijek fresh data od Supabase). Calibracija = audit ratio cache-hit/miss; target: >70% za static, <5% za API.

3. **Causal vs correlational**: čisto kauzal — `IF (offline AND form_submitted) THEN write to IDB outbox; IF (online AND outbox_non_empty) THEN POST sync`. Bez ML, bez probabilistic sync. Determinističan FSM.

4. **Failure mode taxonomy**: identificirano 5 failure modova:
   - Offline submit + permanent offline (timeout 7 dana) → korisnik dobije UI warning, može retry ili discard
   - Sync u tijeku, browser zatvoren mid-flight → IDB outbox entry ostaje, retry pri sljedećem load-u (idempotent)
   - Server returns 4xx (npr. plan downgrade, schema invalid) → entry ostaje u IDB s `last_error` field; korisnik vidi error u UI-ju
   - Server returns 5xx → exponential backoff retry (max 5 attempts, then alert)
   - Concurrent sync 2 device-a od istog korisnika → server UNIQUE on client_event_id; second device dobije prvi response (idempotent)
